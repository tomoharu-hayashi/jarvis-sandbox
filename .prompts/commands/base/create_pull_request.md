# プルリクエスト作成

以下の指示に従ってプルリクエストを作成してください。

## 注意事項

- マージなどの重要な操作は禁止
- 前提を把握するため、コーディングルールを参照する
- ghコマンドを使用してGitHubを操作する

## プルリクエスト作成プロセス

1. Github ProjectsとGitのworking stateを参照して、行った内容を振り返る
2. ブランチが切られていなければ、簡潔な英語で命名しブランチを作成
3. 洗練されたコミットを作成（下記ルール参照）
4. **関連Issueを検索** — `gh issue list --state open` で Open Issue を確認し、このPRで解決するIssueがあれば番号をメモ
5. 以下のフォーマットに従いPRを作成

## コミットメッセージのルール

### Prefix

- `feat:` — 新機能（アプリの直接の機能のみ）
- `fix:` — バグ修正
- `refactor:` — リファクタリング
- `docs:` — ドキュメント
- `test:` — テスト
- `chore:` — ビルド、CI、依存関係など
- `style:` — フォーマット、セミコロンなど

### フォーマット

```
<prefix>: <日本語で簡潔な説明>
```

### コミット分割の指針

- 意思決定ごとに適切な範囲で分割
- 1コミット = 1つの論理的な変更
- レビューしやすい粒度を意識

### 例

```
feat: ユーザー認証機能を追加
fix: ログイン時のバリデーションエラーを修正
refactor: API呼び出しを共通化
docs: READMEにセットアップ手順を追加
test: 認証フローのユニットテストを追加
chore: ESLint設定を更新
```

## PRフォーマット

```markdown
Closes #<issue番号>  <!-- 関連Issueがある場合のみ。複数あれば Closes #1, Closes #2 -->

## 概要
<!-- このPRで「何を」「なぜ」やるのかを1〜3行で -->

## コミット
<!-- コミット順に簡潔に箇条書きで記載 -->
例:
- feat: ユーザー登録フォームを追加 a1bd9b6
- fix: メールアドレスのバリデーションを修正 b2ce0c7
- refactor: フォームコンポーネントを共通化 c3df1d8
- test: 登録フローのE2Eテストを追加 d4eg2e9

## メモ
<!-- レビュー時に見てほしい観点や、将来の自分へのメモ -->
例:
- TODO: 他のAPIにも同様のバリデーションを追加する必要あり（別PR予定）
- バリデーションエラーのメッセージは暫定なのでUX面で改善余地あり

## チェックリスト
<!-- チェックリストは箇条書きで記載 -->
- [ ] lint / type check 済み
- [ ] テスト追加・修正済み
- [ ] 手動での動作確認済み
<!-- その他に追加すべきチェックリストがあれば記載 -->

## リソース
<!-- 資料, デザイン, 画像など -->
```
